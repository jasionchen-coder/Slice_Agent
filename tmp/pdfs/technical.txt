直播切片 Agent 技术实现文档
1. 文档目标
本文档用于指导开发一个直播回放自动切片 Agent 系统。


系统需要完成以下核心链路：



 直播回放视频上传
  ↓
 视频信息解析
  ↓
 音频提取
  ↓
 语音转文字
  ↓
 文字清洗与分段
  ↓
 大模型内容分析
  ↓
 生成切片方案
  ↓
 视频自动切割
  ↓
 前端预览与导出


本系统的核心目标不是简单切视频。系统的核心是：先理解直播内容，再根据内容价值生成可发布的视频切
片。




2. 系统整体架构
2.1 架构说明
系统采用前后端分离 + 异步任务队列 + AI Pipeline 的架构。


前端负责视频上传、任务创建、任务进度展示、切片预览和下载。


后端负责文件管理、任务调度、音频提取、ASR 转写、大模型分析、视频切割和数据存储。


异步任务队列负责处理耗时任务。直播视频一般较大，所以不能用同步接口直接处理完整流程。

5. 核心处理流程设计
5.1 任务创建流程

 用户上传视频
  ↓
 前端请求后端创建任务
  ↓
 后端保存视频文件
  ↓
 后端解析视频基础信息
  ↓
 后端创建 task 记录
  ↓
 后端投递异步任务
  ↓
 前端跳转任务进度页




                                            6
任务创建成功后，后端只返回任务 ID。后续处理全部交给 Worker。




5.2 Worker Pipeline 流程

 start_video_clip_task(task_id)
   ↓
 1. 获取任务信息
   ↓
 2. 更新状态为 extracting_audio
   ↓
 3. 用 FFmpeg 提取音频
   ↓
 4. 更新状态为 transcribing
   ↓
 5. 调用 ASR 生成带时间戳文字稿
   ↓
 6. 更新状态为 cleaning_transcript
   ↓
 7. 清洗文字稿并做语义分段
   ↓
 8. 更新状态为 analyzing_content
   ↓
 9. 调用 LLM 分析话题和高价值片段
   ↓
 10. 更新状态为 generating_clips
   ↓
 11. 生成最终切片方案
   ↓
 12. 更新状态为 cutting_video
   ↓
 13. 用 FFmpeg 批量切割视频
   ↓
 14. 保存切片文件和元数据
   ↓
 15. 更新任务状态为 completed




5.3 任务失败流程
任何步骤失败后，系统都要进入统一异常处理。



 处理步骤失败
  ↓
 捕获错误
  ↓
 记录 error_message
  ↓




                                  7
 记录 failed_stage
  ↓
 更新任务状态为 failed
  ↓
 前端展示失败原因
  ↓

系统不能让任务一直停留在处理中状态。

9. 视频处理设计
9.1 视频信息解析
系统需要在任务创建后解析视频基础信息。




                                       15
需要解析的信息：


   1. 视频时长。
   2. 视频大小。
   3. 视频格式。
   4. 视频分辨率。
   5. 视频编码。
   6. 是否存在音频轨道。

建议使用 FFprobe 完成解析。




9.2 音频提取
系统从原视频中提取音频。音频格式建议统一转成 wav。


推荐参数：



 格式：wav
 采样率：16000 Hz
 声道：mono


这样可以减少 ASR 处理成本。




9.3 长音频切分
如果视频时长较长，系统需要先切分音频，再分段进行 ASR。


建议规则：



 每段音频长度：5 分钟到 10 分钟
 相邻音频重叠：1 秒到 3 秒


音频重叠可以减少边界处识别丢字的问题。




9.4 视频切割
系统根据切片方案中的 start_time 和 end_time 进行视频切割。


切割策略：


   1. 默认使用原视频作为输入。
   2. 每个切片独立处理。
   3. 单个切片失败不能影响其他切片。
   4. 切片文件需要保存到对象存储。




                                 16
      5. 切片完成后需要更新 clips 表。

切割前需要校验：


      1. start_time 必须大于等于 0。
      2. end_time 必须小于等于视频总时长。
      3. start_time 必须小于 end_time。
      4. duration 必须在配置范围内。




10. ASR 语音识别设计
10.1 ASR 输入
ASR 输入是音频文件。



 {
     "task_id": "task_001",
     "audio_path": "tasks/task_001/audio/audio.wav",
     "language": "zh"
 }



10.2 ASR 输出
ASR 必须输出带时间戳的结果。



 {
     "segments": [
       {
         "start_time": 12.5,
         "end_time": 18.2,
         "speaker": "speaker_1",
         "text": "今天我们主要讲一下这个产品的核心卖点。"
       }
     ]
 }



10.3 ASR 结果要求
      1. 每个文本片段必须有 start_time。
      2. 每个文本片段必须有 end_time。
      3. 文本片段必须按时间升序排列。
      4. 如果不能区分说话人，speaker 可以为空。
      5. ASR 原始结果需要保存，方便后续排查问题。

11. 文字清洗与分段设计
11.1 清洗目标
ASR 结果通常存在口语词、重复词和断句问题。系统需要做基础清洗。


清洗内容：


  1. 去除过多的语气词。
  2. 合并过短文本。
  3. 修正常见识别错误。
  4. 保留原始时间戳。
  5. 保留原始文本备查。




11.2 分段规则
系统需要把完整文字稿拆成适合大模型处理的块。


建议采用两层分段：


第一层：时间窗口分段


 每 3 分钟到 5 分钟作为一个基础窗口
 相邻窗口保留 10 秒到 20 秒重叠


第二层：语义分段

在时间窗口内继续判断话题边界。


分段依据包括：


  1. 话题变化。
  2. 商品变化。
  3. 问答变化。
  4. 直播环节变化。
  5. 明显转场话术。




12. LLM 内容分析设计
12.1 大模型调用目标
大模型主要完成两个任务：


  1. 识别直播内容结构。




                         18
   2. 生成可切片的视频片段方案。

不建议让大模型直接处理完整长视频文本。系统应该先分段，再合并分析结果。




12.2 LLM Pipeline

 清洗后的文字稿
   ↓
 按时间窗口分块
   ↓

12.5 LLM 输出校验
大模型输出不能直接入库。系统必须做结构校验。


校验规则：


      1. JSON 必须能被解析。
      2. clips 必须是数组。
      3. start_time 必须是数字。
      4. end_time 必须是数字。
      5. start_time 必须小于 end_time。
      6. duration 必须符合系统配置。
      7. title 不能为空。
      8. score 必须在 0 到 100 之间。
      9. risk_level 必须是 low、medium、high 之一。
     10. high 风险切片默认过滤。

如果校验失败，系统可以执行以下动作：



 第一次失败：尝试 JSON 修复
 第二次失败：重新请求 LLM
 第三次失败：标记当前阶段失败

21. 核心验收标准
21.1 主链路验收
   1. 用户可以上传直播回放视频。
   2. 系统可以成功创建任务。
   3. 系统可以提取音频。
   4. 系统可以生成带时间戳的文字稿。
   5. 系统可以调用大模型生成切片方案。
   6. 系统可以按照时间戳切割视频。
   7. 用户可以在前端预览切片。
   8. 用户可以下载切片。




21.2 数据验收
   1. tasks 表有完整任务记录。
   2. transcript_segments 表有带时间戳的文字片段。
   3. clips 表有最终切片记录。
   4. 每个 clip 都有 start_time 和 end_time。
   5. 每个 clip 都有 title、summary、reason 和 score。
   6. 切片文件可以通过 clip_url 播放。




                                         29
21.3 异常验收
  1. 视频格式错误时，系统能提示用户。
  2. 音频提取失败时，任务会进入 failed 状态。
  3. ASR 失败时，系统会记录失败原因。
  4. LLM 输出错误时，系统会重试或修复。
  5. 单个切片切割失败时，不影响其他切片。
  6. 用户可以对失败任务执行重试。




22. 开发注意事项
  1. 不要把完整长视频处理放在同步接口里。
   视频处理必须走异步任务。


  2. 不要只保存最终切片。
     系统必须保存中间结果。中间结果包括音频、文字稿、LLM 输出和切片方案。


  3. 不要直接信任大模型输出。
     系统必须做 JSON 解析、字段校验和时间戳校验。


  4. 时间戳统一用秒数存储。
     前端展示时再转换成 00:00:00 格式。


  5. 单个切片失败不能导致整个任务失败。
     系统应该把失败切片标记为 failed。其他切片继续生成。


  6. 任务状态必须及时更新。
     用户需要知道系统处理到了哪一步。


  7. 原视频和切片文件需要使用对象存储。
     不建议直接放在 API Server 本地磁盘。

24. 总结
直播切片 Agent 的技术核心是一个异步 AI 视频处理流水线。


系统最重要的三个技术点是：


  1. ASR 必须生成准确的时间戳。
  2. LLM 必须输出结构化切片方案。
  3. FFmpeg 必须根据时间戳准确切割视频。

MVP 阶段应该先保证主链路跑通。也就是：



 视频上传 → 音频提取 → ASR 转写 → LLM 分析 → 视频切割 → 前端预览下载


主链路稳定后，再做字幕、封面、自动发布和实时切片。
